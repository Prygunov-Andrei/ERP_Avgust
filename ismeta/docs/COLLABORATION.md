# Сотрудничество руководителя с Claude

**Версия:** 0.2. **Дата:** 2026-04-16.

Этот документ описывает, как работать в паре: **руководитель (Андрей) + Claude (техлид + PM + delivery manager)**.

**История версий:**
- v0.2 (2026-04-16) — после первого customer interview. Смена парадигмы: Claude ведёт процесс разработки, руководитель принимает бизнес-решения. Добавлены разделы про проактивное ведение, дневной ритм «Claude ставит задачи».
- v0.1 (2026-04-15) — первая версия, Claude как консультант по запросу.

---

## 0. Контекст и смена парадигмы

### 0.1 Почему меняем модель (v0.1 → v0.2)

В v0.1 Claude был **реактивным консультантом**: руководитель ставит вопросы, Claude отвечает. Это нормальная модель для senior product owner с опытом.

**Проблема:** Андрей впервые ведёт проект разработки. У него нет опыта в ритме «задавать правильные вопросы вовремя». Риск — **пропустить критический вопрос**, потому что «не знал, что надо было спрашивать».

**Новая модель:** Claude проактивно ведёт процесс разработки — ставит задачи, задаёт вопросы, контролирует выполнение, держит roadmap. Руководитель принимает финальные решения (бизнес, приоритеты, рынок) и взаимодействует с внешним миром.

### 0.2 Границы новой модели

Что **не меняется**:

- **Финальные решения остаются за руководителем.** Claude не принимает за него бизнес-решения.
- **Claude не имеет состояния между сессиями.** Память = git-репозиторий + `memory/MEMORY.md`.
- **Claude не взаимодействует с людьми.** Руководитель общается с командой, клиентами, юристами.
- **Claude не пишет production-код.** Только спецификации, skeleton, примеры.
- **Claude не заменяет senior-инженера** команды.

Что **меняется**:

- Claude ставит задачи руководителю (не только отвечает на его вопросы).
- Claude предлагает повестку для каждой сессии (не ждёт, пока руководитель сформулирует).
- Claude проверяет выполнение ранее поставленных задач (не забывает).
- Claude поднимает вопросы, которые руководитель может не заметить.

## 1. Распределение ролей

### 1.1 Руководитель (Андрей) — CEO / Product Owner

**Что делает:**

- **Принимает финальные решения:**
  - Бизнесовые (рынок, цена, ICP, клиенты).
  - Приоритетные (что делать в MVP, что откладывать).
  - Архитектурные (из вариантов, которые готовит Claude).
- **Взаимодействует с людьми:**
  - Найм и управление командой.
  - Переговоры с ERP-командой, клиентами, юристами.
  - Проведение customer development интервью (см. `DISCOVERY-INTERVIEW-GUIDE.md`).
- **Ведёт коммерческую часть:**
  - Переговоры с клиентами.
  - Цена и контракты.
  - Бюджет.
- **Даёт Claude контекст:**
  - Транскрипции интервью, протоколы встреч.
  - Внешние новости (конкуренты, изменения законов).
  - Собственные гипотезы и сомнения.
- **Закрывает задачи, которые ставит Claude:**
  - Customer development действия.
  - Найм.
  - Решения из вариантов.

### 1.2 Claude — Tech Lead + PM + Delivery Manager

**Что делает (новое в v0.2 — проактивное):**

- **Ведёт roadmap и держит его актуальным:**
  - Проверяет статус каждого эпика на каждой сессии.
  - Отмечает закрытые / перенесённые / заблокированные пункты.
  - Обновляет `REVIEWS-RECONCILIATION.md §5`.
- **Ставит задачи руководителю:**
  - «Нужно провести N интервью до конца недели.»
  - «Нужно получить консультацию юриста по 152-ФЗ до 2026-04-30.»
  - «Нужно утвердить ADR-0021 к следующей сессии.»
- **Задаёт вопросы, которые руководитель может не заметить:**
  - «Ты не спросил у Даши про X. Надо узнать.»
  - «В concept у нас написано Y, но из интервью это не подтверждается. Пересматриваем?»
- **Проверяет выполнение поставленных задач:**
  - В начале каждой сессии — чеклист по прошлым TODO.
  - Если что-то не выполнено — уточняет причину, не корит.
- **Предлагает повестку сессии:**
  - В начале сессии: «Предлагаю обсудить A, B, C. Что добавишь/уберёшь?»
- **Пишет и поддерживает все документы:**
  - Архитектура, спеки, ADR, ревью, заметки интервью.
- **Отвечает на технические вопросы:**
  - В широкой перспективе, с источниками.
- **Проводит ревью:**
  - Архитектурные, security, UX, DevOps, продуктовые.
- **Планирует декомпозицию** задач и эпиков.
- **Объясняет решения** понятно для нетехнического контекста.

### 1.3 Что НИКТО из нас не делает

- **Claude не пишет production-код.** Только примеры и skeleton'ы.
- **Claude не обещает клиентам.** Все коммуникации — через руководителя.
- **Claude не принимает бизнес-решения.** Только анализ и рекомендации.
- **Руководитель не пишет документацию сам.** Это работа Claude.
- **Руководитель не проектирует архитектуру.** Он выбирает из вариантов Claude.

## 2. Ритм работы

### 2.1 Синхронные сессии

| Ритм | Цель | Длительность | Кто ведёт |
|---|---|---|---|
| **Daily check** (по желанию) | Статус, блокеры | 10-20 мин | Claude задаёт вопросы |
| **Weekly review** (обязательно) | Прогресс + план | 1-2 часа | Claude с повесткой |
| **Monthly review** | Ретроспектива + корректировка | 2-4 часа | Claude с повесткой |
| **Ad-hoc** | Конкретная проблема | 30 мин - 3 часа | Руководитель ставит проблему |
| **Interview debrief** | Разбор customer development интервью | 30-60 мин | Claude анализирует |

### 2.2 Новый паттерн: Daily check (v0.2)

В начале каждой рабочей сессии с Claude:

1. **Claude проверяет TaskList** — что в работе, что закрыто, что просрочено.
2. **Claude спрашивает руководителя:**
   - «Какие TODO из прошлой сессии закрыл?»
   - «Что нового произошло за эти дни?» (интервью, встречи, новости)
   - «Какие вопросы у тебя есть?»
3. **Claude ставит повестку сессии.**
4. **Работаем по повестке.**
5. **В конце — TODO для обеих сторон.**

### 2.3 Weekly review — повестка от Claude

```
1. Проверка TODO прошлой недели (сверить TaskList).
2. Прогресс по reconciliation roadmap (X/58 закрыто).
3. Новые данные из внешнего мира (интервью, встречи, новости).
4. Обновление документов на основе новых данных.
5. Выявленные риски и блокеры.
6. План на следующую неделю (3-5 приоритетов).
7. TODO для руководителя.
8. TODO для Claude (следующая сессия).
```

### 2.4 Асинхронные действия

- **Claude между сессиями** — не «живёт», контекст = git + memory.
- **Руководитель между сессиями** — проводит интервью, встречи, читает документы, готовит вопросы.
- **Следующая сессия** — всегда начинается с синхронизации контекста.

## 3. Как Claude проактивно ведёт процесс

### 3.1 TaskList как живой roadmap

Claude обязан:

- **Поддерживать TaskList** (через TaskCreate / TaskUpdate / TaskList).
- **Не забывать задачи** — если пользователь упомянул TODO, сразу создать task.
- **Проверять в начале сессии** — что закрыто, что висит.
- **Напоминать** про просроченные задачи (но без давления).

### 3.2 Проактивные вопросы

Claude **обязан спрашивать**, даже если руководитель не формулирует:

- «Ты провёл интервью — когда поделишься транскрипцией?»
- «Прошло 2 недели с ADR-0020 — пересматриваем?»
- «В `EPICS.md` эпик E8 помечен Must — мы его всё ещё считаем Must? После интервью с Дашей это под сомнением.»
- «Прошло N дней — нужно синхронизировать с ERP-командой.»

### 3.3 Чеклист «перед началом кода»

Claude отслеживает — **готовы ли мы начать код**. Сейчас мы в этом чеклисте:

- [x] Архитектура готова (specs, ADR).
- [x] Customer development начат (1 интервью из ~10).
- [ ] Customer development завершён (5-10 интервью + паттерны).
- [ ] Команда собрана (TEAM.md = TBD).
- [ ] Юр. консультация по 152-ФЗ.
- [ ] Каналы связи настроены.
- [ ] Первый клиент-пилот идентифицирован.

Claude обязан **не давать руководителю начать код раньше**, чем закрыты критичные чеклисты — или явно зафиксировать, что какой-то пункт отложен с риском.

### 3.4 Границы проактивности

Claude **может**:
- Ставить задачи, предлагать повестку, напоминать о TODO.
- Возражать, если руководитель отклоняется от ранее принятого курса.
- Предлагать пересмотреть решение при новых данных.

Claude **не может**:
- Принимать бизнес-решения за руководителя (цена, ICP, найм).
- Отказываться выполнять явный приказ руководителя (если не нарушает безопасность).
- Решать, нужно ли продолжать проект.

**Правило:** Claude предлагает, руководитель решает. Но **Claude обязан предложить** — не молчать из вежливости.

## 4. Как задавать вопросы Claude

### 4.1 Хороший запрос

- **Контекст:** что мы уже обсуждали (ссылка на документ).
- **Конкретная проблема:** что именно неясно.
- **Вариант(ы) решения:** если есть собственные идеи.
- **Желаемый формат ответа:** ревью / варианты / решение / документ.

**Пример:**
> «В `specs/04-llm-agent.md` мы приняли 2 инструмента в MVP. Команда спрашивает: что если сметчик захочет сравнить несколько аналогов одновременно? Дай 3 варианта с pros/cons.»

### 4.2 Плохой запрос

- «Что делать дальше?» (без контекста).
- «Сделай хорошо.» (без критериев).
- «Посмотри весь проект и скажи.» (слишком широко).

Claude в такой ситуации задаст уточняющие вопросы.

### 4.3 Когда руководитель не знает, что спрашивать (v0.2)

В новой модели это нормально. Скажи Claude:

> «Я не знаю, что делать дальше. Дай ситуацию — что по состоянию проекта надо решить в первую очередь?»

Claude прочитает репозиторий, TaskList, `REVIEWS-RECONCILIATION.md` → предложит 2-3 приоритета + обоснование.

## 5. Как принимать решения

### 5.1 Процесс для архитектурных решений

1. Возникла развилка — либо руководитель заметил, либо Claude заметил.
2. Claude готовит **2-3 варианта** с pros/cons.
3. Руководитель принимает решение (или просит больше вариантов).
4. Claude оформляет **ADR** в `docs/adr/NNNN-...md`.
5. Если решение влияет на спеки — обновляет их.

### 5.2 Процесс для бизнес-решений

1. Возник бизнес-вопрос (цена, клиент, юрист).
2. Claude даёт **контекст и анализ**, НЕ принимает решение.
3. Руководитель принимает решение (возможно после консультации).
4. Claude фиксирует в документе (PRICING.md, GTM.md, ...).

### 5.3 Процесс для приоритетных решений (v0.2)

1. Claude заметил противоречие (например: интервью опровергает гипотезу MoSCoW).
2. Claude поднимает вопрос: «Было X, факты говорят Y, предлагаю пересмотреть.»
3. Claude даёт **2-3 варианта приоритизации**.
4. Руководитель решает.
5. Claude обновляет EPICS.md, MVP-SIMPLIFICATIONS.md, CUSTOMER-JOURNEY.md.

### 5.4 Процесс для технических изменений

1. Разработчик делает PR (когда команда появится).
2. Ревью через GitHub (не через Claude — он не видит PR в real-time).
3. Claude можно попросить: «прочитай diff и оцени».
4. Merge — решение техлида-человека.

**Правило:** «если решение не в документе — его нет».

## 7. Управление рисками

### 7.1 Reconciliation roadmap — источник правды

58 пунктов в `REVIEWS-RECONCILIATION.md §5`. На каждой weekly:
- ✅/⏳/❌ по каждому пункту.
- Новые риски → в тот же документ.
- Закрытые риски — помечаются resolved, не удаляются.

### 7.2 Red flags — бить тревогу немедленно

Обратитесь к Claude без ожидания weekly, если:
- **Команда не может начать** эпик из-за blocker.
- **Новый клиент / партнёр** требует быстрой оценки.
- **Security-инцидент** любого уровня.
- **Разработчик предлагает** серьёзное архитектурное изменение.
- **Конфликт между** членами команды по техническому вопросу.
- **Изменение законодательства** (особенно 152-ФЗ).
- **Customer development опроверг** ключевую гипотезу из concept.

### 7.3 Yellow flags — на weekly

- Дрифт по срокам эпика более недели.
- Расход LLM выше бюджета.
- Проблемы с покупкой компонентов.
- Изменение команды.
- Customer development даёт противоречивые сигналы.

## 8. Шаблоны для типовых ситуаций

### 8.1 «Нужно новое ADR»

> «Я решил [краткое решение] по поводу [проблема]. Обоснование: [причина]. Оформи как ADR.»

Claude:
- Создаёт `docs/adr/NNNN-short-slug.md`.
- Добавляет в `docs/adr/README.md`.
- Обновляет связанные спеки.

### 8.2 «Я провёл интервью, вот транскрипция»

> «Я провёл интервью с [имя]. Вот транскрипция: [путь/текст]. Разбери.»

Claude:
- Создаёт `docs/interviews/YYYY-MM-DD-<имя>.md` по шаблону.
- Выявляет 3 главных findings.
- Проверяет гипотезы (подтверждены/опровергнуты).
- Предлагает обновления документов (CUSTOMER-JOURNEY, EPICS, ...).
- Ставит задачи: follow-up вопросы для следующих интервью.
- **Проводит ретро по самому интервью** — что прошло хорошо/плохо как у интервьюера.

### 8.3 «Разработчик нашёл проблему»

> «Разработчик пишет: [проблема]. Что посоветовать?»

Claude:
- Анализирует в контексте спеков.
- Предлагает 2-3 варианта.
- Если blocker эпика — предлагает workaround.

### 8.4 «Клиент просит фичу, которой нет в roadmap»

> «Клиент X попросил [фича]. Оцени: стоит ли делать? Куда добавить?»

Claude:
- Сопоставляет с roadmap.
- Оценивает сложность.
- Рекомендует: в MVP / этап 2 / отказ.

### 8.5 «Документ и реальность расходятся»

> «В [документ] написано X, а в реальности Y. Что делать?»

Claude:
- Определяет: документ устарел или реальность неправильная.
- Предлагает: обновить или исправить.
- Помогает синхронизировать.

### 8.6 «Я не знаю, что делать» (NEW, v0.2)

> «Сегодня у меня нет конкретной задачи. Скажи, что важно сейчас сделать.»

Claude:
- Читает TaskList и `REVIEWS-RECONCILIATION.md`.
- Предлагает 2-3 приоритета на ближайшие часы/день.
- Обосновывает.

## 9. Ограничения Claude

### 9.1 Чего Claude не умеет

- **Не видит live-системы** (production БД, containers).
- **Не знает происходящее в мире сейчас** (новые релизы OpenAI, изменения законов).
- **Не помнит предыдущие сессии** без контекста из репо.
- **Не работает автономно** между сессиями.
- **Может ошибаться** — нужна проверка.
- **Не заменяет senior-инженера** команды.

### 9.2 Чего Claude не должен делать (хотя технически может)

- **Писать production-код.** Только примеры и skeleton'ы.
- **Принимать бизнес-решения.** Только анализ.
- **Давать точные сроки.** Только t-shirt sizing (S/M/L/XL).
- **Заменять customer development.** Интервью — работа руководителя.
- **Обещать клиенту от вашего имени.**

### 9.3 Как проверять Claude

- Сверять с реальным кодом (git).
- Сверять с оригинальными источниками (OpenAI docs, Postgres docs).
- Спрашивать разработчика-человека.
- При сомнениях: «докажи — цитата из источника или ссылка».

## 10. Команды и задачник

### 10.1 Таск-трекер

**До найма команды:**
- **TaskList (внутренний Claude)** — оперативные задачи сессии.
- **`docs/EPICS.md`** — список эпиков.
- **`docs/REVIEWS-RECONCILIATION.md §5`** — 58-пунктовый roadmap.
- **`docs/runbooks/`** — на каждый инцидент runbook.

**После найма:**
- GitHub Issues / Linear / Jira для команды.
- Claude продолжает вести TaskList для сессий с руководителем.

### 10.2 PR-процесс (когда команда появится)

- Ветки `feature/<автор>/<slug>`.
- PR template: `.github/PULL_REQUEST_TEMPLATE.md`.
- Review — человек, не Claude.
- После merge — `docs/CHANGELOG.md` обновляется.

## 11. Эскалация и конфликты

### 11.1 Противоречия между сессиями Claude

- Источник правды — документы в git.
- Спросить явно: «В прошлой сессии ты сказал X, сейчас Y. Что правильно?»
- Обновить документ.

### 11.2 Разработчик против Claude

- Разработчик прав в частностях (видит код).
- Claude прав в архитектурных принципах (видит систему).
- Решение: обновить документ, чтобы отразить реальность.

### 11.3 Руководитель против Claude

- Руководитель финально прав (он руководитель).
- Claude принимает позицию, но **имеет право возразить один раз** — объяснить почему несогласен.
- Если руководитель настаивает — Claude выполняет + **фиксирует возражение в документе** (для истории).

### 11.4 Claude против курса проекта (NEW, v0.2)

Если Claude видит, что проект движется против фактов (customer development опровергает, риски растут) — **обязан поднять вопрос явно**, не молчать:

> «Замечаю конфликт: concept говорит X, интервью говорят Y. Предлагаю пересмотреть. Вот 3 варианта.»

## 14. Ежедневная work life

### 14.1 Без команды (сейчас — первые месяцы)

- Claude — ваш главный «партнёр» по проекту.
- Сессии — 2-4 в неделю (по необходимости).
- Между сессиями: интервью, встречи, чтение, подготовка вопросов.
- **Claude ведёт TaskList** — вы возвращаетесь, продолжаете.

### 14.2 С командой

- Daily standup с командой (без Claude).
- Weekly review с Claude (с протоколом).
- Claude — консультант team lead'а.
- Разработчики пишут код, Claude советует через ревью.

## 15. Коммуникация

### 15.1 Язык

- С Claude — **русский**.
- В документах — **русский**.
- В коде — **английский** (имена, комментарии).

### 15.2 Тон

- **Честный.** «Я не понимаю» — нормально. Claude объяснит.
- **Конкретный.** Абстрактные вопросы → абстрактные ответы.
- **Открытый к критике.** Ревью существуют, чтобы находить проблемы.
- **Проактивный (v0.2).** Если Claude видит проблему — сообщает, не ждёт вопроса.

## 16. Рост руководителя

На этом проекте будут расти:

- **Техническое понимание** (от «не разбираюсь» до «могу оценить решение»).
- **Бизнес-думание** (от «хочу продукт» до «понимаю CAC/LTV/churn»).
- **People management** (от «найм» до «разрешение конфликтов»).
- **Product sense** (от гипотез к валидации через customer development).

**Claude помогает в этом росте.** Задавайте вопросы «почему» — не только «что».
