# Открытые вопросы — UX для работы с Счетами и КП в ERP

**Статус:** отложено. Возвращаемся когда будет capacity для proper UX-дизайна.

**Контекст:** 2026-04-22 при проектировании E16/E17 (backend-расширение Recognition на Invoice+Quote) всплыли вопросы про workflow. PO подтвердил — это отдельная сложная тема. Backend работу продолжаем (Recognition distinguishes форматы документов), но UI/workflow не проектируем пока не решён продуктовый дизайн.

---

## Ответы PO на 2026-04-22

1. **Где в ERP пользователь загружает xlsx/PDF КП?**
   PO: «по всей вероятности нигде пока нет». → UI Загрузки КП отсутствует. Recognition работает через API только.

2. **Workflow после распознавания КП** (просто чтение / сравнение с конкурентами / конвертация в цены сметы)?
   PO: «очень сложные вопросы! Не смогу сейчас ответить — это очень сложная отдельная тема».
   → **Открытый вопрос.** Нужен отдельный продуктовый спринт.

3. **Multi-currency** (Бриз EUR/CNY/RUB)?
   PO: «в смете точно только рубли». → Принято: `QuoteItem.currency = "RUB"` жёстко для смет. Если КП в EUR/CNY — конвертация неявная (по текущему курсу? на дату КП? — **открытый вопрос** если КП не в RUB уже).

4. **Group-headers ПВ1/ПВ2/К1.1** — как отображать в смете?
   PO: «не могу ответить вообще». → **Открытый вопрос.**

---

## Что этого значит для работы

- **Recognition backend** (E15.05 SpecParser, E16 Invoice, E17 Quote xlsx) — это **расширение возможностей парсинга**. Backend-only, готовится впрок.
- **Применение** (как результат Recognition отображается / используется / конвертируется в сущности ERP) — **отдельная большая тема**, ждёт продуктового решения.
- **Не проектируем UI Поставщики/Счета/КП** пока workflow не решён на уровне бизнес-процессов.

---

## Следующие шаги когда вернёмся к этой теме

1. **UX-исследование** — опросить менеджеров проектов: как они сейчас работают с КП от поставщиков? Сравнивают в Excel вручную? В email?
2. **Продуктовая сессия** — прорисовать flow:
   - Поступление КП → ячейка хранения.
   - Сравнение 3-5 КП по одной потребности.
   - Выбор поставщика → формирование заявки на закупку / Счёта.
   - Привязка к смете (если конкретный item в КП матчится со строкой сметы).
3. **Архитектурное решение** — КП это отдельная сущность `Proposal` или embedded в `Estimate`/`Order`?
4. **ТЗ UI** — UI-09 «Загрузка КП», UI-10 «Сравнение КП», UI-11 «Конвертация КП в заказ».

Пока всё это отложено. Backend готов принимать КП в любом формате и возвращать структурированную схему — когда workflow будет решён, UI сможет опираться на готовое API.
