### 0.1 Почему такая команда

- Стартовая конфигурация для соло-основателя, который хочет быстро начать без бюджета на найм.
- Подтверждено: быстрый старт кода (параллельно с customer development).
- Риск: двойная AI-петля без human-валидации критична. Андрей обязан запускать код руками.

### 0.2 Открытые слоты (приоритет найма)

| Роль | Когда берём | Почему |
|---|---|---|
| Live QA-сметчик (dogfood) | Этап 1, неделя 2 | один из участников интервью (Даша / Оля / Катя / внешний) тестирует прототип |

## 1. Роли и ответственности

### 1.1 Андрей (CEO/PO)

**Делает:**
- Бизнес-решения: рынок, цена, ICP, приоритеты.
- Customer development: проводит интервью, собирает транскрипции.
- Найм: когда команда расширится.
- **Human in the loop** для AI-команды — запускает код, проверяет что работает.
- Передаёт задачи между Tech Lead и Developer (я ↔ Петя).
- Приносит Tech Lead'у реальные факты: логи, скриншоты, ошибки.

**Не делает:**
- Не пишет код.
- Не пишет документацию (это работа Tech Lead).
- Не проектирует архитектуру сам (выбирает из вариантов).

### 1.2 Tech Lead (Claude session A, Андрей общается в этом окне)

**Делает:**
- Ведёт процесс разработки (roadmap, TaskList, повестка сессий).
- Проектирует архитектуру (2-3 варианта → Андрей выбирает).
- Пишет документацию (specs, ADR, ревью, заметки интервью).
- Ставит задачи Пете через Андрея.
- Проводит code review: читает diff / ветку Пети, пишет правки.
- Анализирует customer development findings.

**Не делает:**
- Не пишет production-код.
- Не контактирует напрямую с Петей (всегда через Андрея).
- Не работает в feature-ветках Пети одновременно с ним.

### 1.3 Петя (Claude session B, AI-разработчик)

**Делает:**
- Читает задачу от Tech Lead.
- Пишет код, тесты, миграции в feature-ветке.
- Делает commits с Conventional Commits форматом.
- Запускает локально (`make ismeta-ci-local`).
- Возвращает отчёт Андрею: что сделано, что запустил, что не работает.

**Не делает:**
- Не принимает архитектурные решения без Tech Lead.
- Не меняет documentation без отдельной задачи.
- Не мержит в `main` — только открывает PR.
- Не работает в ветках других (main, docs).
- Не говорит «работает» без реального запуска с логами.

## 2. Git workflow для AI-команды

### 2.1 Структура веток

```
main                       — документация, стабильный код (Tech Lead)
feature/petya/e1-infra     — Петя работает тут
feature/petya/e2-auth      — следующая задача Пети
docs/<slug>                — Tech Lead работает с документами
```

**Правило:** Петя **никогда** не пушит в `main` напрямую. Всегда `feature/petya/<slug>` → PR → ревью Tech Lead → merge.

### 2.2 Race condition prevention (КРИТИЧНО)

**Железное правило:** в один момент времени только один AI пишет в репозиторий.

Конкретно:
- **Петя работает** → Tech Lead **только читает** (Read/Grep, никаких Edit/Write).
- **Tech Lead пишет документы** → Петя **не трогает репо** (Андрей ставит на паузу).
- **Передача эстафеты** — через Андрея: «Петя закончил задачу X, код в ветке Y, теперь пишу в main».

Если нарушено — один AI переписывает изменения другого (потеря работы).

### 2.3 Цикл задачи (от постановки до merge)

```
1. Tech Lead формирует промпт с задачей (в сессии с Андреем).
2. Андрей копирует промпт Пете.
3. Петя:
   a. git checkout -b feature/petya/<slug>
   b. пишет код + тесты
   c. запускает локально
   d. commits в ветку (НЕ push, либо push в origin)
   e. возвращает отчёт Андрею
4. Андрей запускает у себя (git checkout ветки + make test).
5. Андрей приносит Tech Lead'у: «ветка feature/petya/X, тесты зелёные, запустил — работает».
6. Tech Lead читает diff, пишет review comments.
7. Андрей передаёт Пете comments.
8. Петя правит → шаг 4.
9. Когда чисто — squash merge в main (делает Андрей или Петя по команде).
```

### 2.4 Коммуникация через Андрея

**Формат передачи Пете (от Tech Lead):**

```
# Задача #N: [название]
## Контекст
[ссылки на specs / ADRs]
## Цель
[что должно получиться]
## Критерии приёмки
- [ ] ...
## Что НЕ делать
- ...
## Deliverable
- файлы: ...
- тесты: ...
- ветка: feature/petya/<slug>
```

**Формат отчёта от Пети (Андрею):**

```
## Задача #N — отчёт
- Ветка: feature/petya/<slug>
- Commits: N
- Тесты: все зелёные / N упали (лог)
- Запускал локально: да/нет
- Что не ясно: ...
- Нужно от Tech Lead: ревью / уточнение / ничего
```

## 3. Human-in-the-loop правила

Ключевая функция Андрея — **проверка что AI не галлюцинирует**.

### 3.1 Обязательные проверки после каждого PR Пети

1. **git checkout** ветки локально.
2. **Запустить** `make ismeta-ci-local` — должно пройти.
3. **Запустить** `make ismeta-test` — все тесты должны быть зелёные.
4. **Открыть** UI / API в браузере (если это фронт/бэк).
5. **Приложить** скриншот или лог в отчёт Tech Lead.

Если **любой** из шагов не прошёл — **не мержим**, возвращаем Пете с логом.

### 3.2 Правило «никто не сказал "работает" без запуска»

- Петя не говорит «работает» — говорит «запустил локально, вот логи».
- Tech Lead не говорит «код хороший» без запуска у Андрея.
- Андрей не говорит «готово» без `make ismeta-ci-local`.

### 3.3 Red flags галлюцинации

Если Андрей видит:
- Петя ссылается на файл, которого нет в репо.
- Петя утверждает что библиотека X работает, а Tech Lead говорит иначе.
- Тесты «все прошли», но Петя не запускал их.
- Петя пишет `# TODO` без сопутствующего изменения.
- Пропал файл, который был вчера.

→ **Пауза. Откат ветки. Анализ с Tech Lead.**

## 4. Daily ритм (рабочий день Андрея)

### 4.1 Утром

1. Session с Tech Lead: что делаем сегодня.
2. Tech Lead выдаёт промпт для Пети.
3. Андрей копирует Пете, Петя начинает.

### 4.2 В течение дня

4. Пока Петя пишет — Андрей занят:
   - Customer development (интервью, рассылки).
   - Встречи с командой / партнёрами.
   - Business tasks (цена, маркетинг, юр).
5. Петя отчитывается — Андрей смотрит логи, запускает.

### 4.3 Вечером

6. Session с Tech Lead: приносит отчёт Пети + свои findings.
7. Tech Lead делает ревью.
8. Если правки — Петя правит утром.
9. Если чисто — merge.

### 4.4 Синхронизация

- Андрей не работает параллельно с Петей в репо.
- Tech Lead не работает с кодом параллельно с Петей.
- Один рабочий поток = один AI пишет.

## 5. Регулярные встречи

| Встреча | Частота | Длительность | Участники |
|---|---|---|---|
| Daily утром (Tech Lead) | ежедневно | 15-30 мин | Андрей + Tech Lead |
| Daily вечером (Tech Lead) | ежедневно | 15-30 мин | Андрей + Tech Lead |
| Weekly review | пн вечером | 1-2 часа | Андрей + Tech Lead |
| Monthly review | последняя пт месяца | 2-3 часа | Андрей + Tech Lead |
| Customer interviews | по мере проведения | 30-60 мин | Андрей + сметчик |
| Interview debrief | после интервью | 30 мин | Андрей + Tech Lead |

**Пети нет на встречах** — он общается только через задачи и отчёты. Это осознанно: AI-агенты не обсуждают, они получают задачу и делают.

## 6. Эскалация

### 6.1 Что требует немедленной реакции (red)

- Петя сделал что-то критично не так (удалил файлы, сломал main).
- Код не запускается у Андрея (а Петя сказал «работает»).
- Обнаружено что AI галлюцинирует (см. §3.3).
- Customer development опровергает ключевую гипотезу.
- У Андрея личный блокер (болезнь, отпуск) — как работает команда без него.

→ Андрей обращается к Tech Lead сразу, вне daily.

### 6.2 Что обсуждаем на weekly (yellow)

- Дрифт по срокам эпика.
- Расход на LLM (считаем обе сессии: Tech Lead + Петя).
- Качество кода Пети падает (много fix-ups).
- Новые insights с customer development.

### 6.3 Когда нужен живой разработчик

**Красные линии — нанимаем human senior:**

- Петя даёт неверный код 3 задачи подряд — архитектурная проблема.
- Tech Lead не может ревьюить сложные детали (например, performance tuning Postgres).
- Появляется production с реальными клиентами (AI не дежурит on-call).
- Security-чувствительный код (JWT rotation, криптография).

## 7. Метрики эффективности команды

Отслеживаем (Tech Lead ведёт в заметках):

| Метрика | Цель MVP | Комментарий |
|---|---|---|
| Velocity Пети (задач в неделю) | 5-10 | зависит от сложности |
| Fix-up rate (правок после ревью) | < 30% | если выше — Tech Lead плохо ставит задачи |
| Broken commits на main | 0 | железное правило |
| Customer interviews done | 5-10 до кода | validation gate |
| Сроки эпика E1 | 7 дней | если больше — эскалация |

## 8. Расходы

- **Claude Opus 4.6 (Tech Lead сессия):** ~20-50 запросов/день × ~$0.50 = **$10-25/день**.
- **Claude Opus 4.6 (Петя сессия):** ~50-100 запросов/день × ~$0.80 = **$40-80/день**.
- **Итого Claude:** **$50-100/день** = **$1500-3000/месяц**.
- Бесплатные альтернативы: прекратить работу с AI или ограничить сессии.

Сравнение: human fullstack в РФ = ₽200-300К/мес (≈$2000-3000). По бюджету сопоставимо, по скорости — быстрее.

### 8.1 Оптимизация расходов

- Петя работает только в активные часы (не круглосуточно).
- Tech Lead использует Explore subagents для поиска (меньше opus-запросов).
- Context window не превышать 70% — своевременно /compact.
