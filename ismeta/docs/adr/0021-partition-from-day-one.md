# ADR-0021. Партиционирование append-heavy таблиц с первой миграции

- **Статус:** Accepted
- **Дата:** 2026-04-16
- **Supersedes:** [ADR-0012](./0012-auditlog-1year.md) (в части партиционирования)
- **Источник:** ревью Пети (онбординг 2026-04-16), `specs/01-data-model.md §8.10`.

## Контекст

Три таблицы растут append-only и достигнут десятков/сотен миллионов строк:

- **EstimateItem** — строки смет. На клиента 50 сметчиков × 10 смет/месяц × 500 строк = 250К/месяц = 3М/год.
- **AuditLog** — история изменений. 10-50 действий на смету/день × 10 сметчиков × 250 рабочих дней = 25К-125К/год на workspace. На коробке 50 сметчиков — до 2М/год.
- **LLMUsage** — журнал LLM-запросов. Зависит от интенсивности matching — оценочно 100К-1М записей/год.

ADR-0012 предложил: «не партиционируем сразу, ставим триггер на 2М строк → тогда партиционируем».

Проблема: партиционирование живой таблицы на 2М+ строк требует **downtime 1-2 часа** (копирование данных в новую структуру + пересоздание индексов + переключение). На production с клиентами это неприемлемо.

При этом добавить partitioning на первой миграции — **1 день работы** + фоновая Celery-задача для создания партиций вперёд.

Противоречие между ADR-0012 и `specs/01-data-model.md §8.10` разрешается в пользу spec'и.

## Решение

**Партиционируем с первой миграции:**

| Таблица | Тип партиции | Ключ | Период |
|---|---|---|---|
| `estimate_item` | HASH | `workspace_id` | фиксированное число партиций (16) |
| `audit_log` | RANGE | `created_at` | monthly |
| `llm_usage` | RANGE | `created_at` | monthly |
| `matching_session_stats` | RANGE | `created_at` | monthly |

Celery-beat task `maintain_partitions` запускается раз в сутки:

- Создаёт партиции на 2 месяца вперёд.
- Удаляет партиции старше retention (AuditLog — 1 год, LLMUsage — 6 месяцев).
- Алертит, если партиция на текущий месяц отсутствует.

## Обоснование

### Почему сразу, а не «когда понадобится»

1. **Миграция 2М+ строк на продакшене** = downtime. Недопустимо при работающих клиентах.
2. **Риск "забыть"**: без триггера на 2М строк легко проскочить границу. Мониторинг размера таблицы — дополнительная нагрузка, которую тоже можно забыть.
3. **Partitioning с первого дня — стандартная практика** для append-heavy таблиц в PostgreSQL.
4. **Стоимость "сейчас" ≈ 1 день** (модели + миграция + Celery-task + unit-тест). Стоимость "потом" ≈ 2-3 дня + downtime + стресс.

### Почему HASH для EstimateItem

- Нет естественного диапазона (дата создания сметы — не горячее поле).
- Равномерное распределение по `workspace_id` = equal load между партициями.
- 16 партиций хватит до ~160М строк без необходимости репартиционирования.
- Фиксированное число → нет нужды в автосоздании партиций для EstimateItem.

### Почему RANGE (monthly) для логов

- Горячие запросы всегда по датам ("покажи что было за последний месяц").
- Partition pruning отрабатывает автоматически при фильтре по `created_at`.
- Удаление старых данных = `DROP PARTITION` (мгновенно), а не `DELETE WHERE` (медленно).
- Monthly — баланс между числом партиций (12/год) и их размером.

## Последствия

### Плюсы

- Linear scale до сотен миллионов строк без переделки.
- Быстрые queries по датам и workspace_id.
- Дешёвое удаление старых данных (DROP PARTITION).
- Нет downtime-миграции «когда понадобится».

### Минусы

- **+1 день работы** на первой миграции (модели + partition DDL + Celery-task).
- **Фоновая задача обязательна.** Если `maintain_partitions` сломалась — записи в новый месяц упадут с ошибкой. Митигация: алерт + retry + health-check.
- **Django ORM не умеет PostgreSQL partitioning из коробки.** Нужен `django-postgres-partitioning` (или raw SQL в RunSQL-миграциях). Доп. зависимость.
- Unit-тесты чуть сложнее: надо проверять что запись попадает в нужную партицию.
- Debugging error'ов в миграциях слегка усложняется — partition DDL более многословный.

### Митигации

- `maintain_partitions` — с алертом при ошибке + health-check эндпоинт `/health/deps` проверяет существование партиции для текущего месяца.
- В `CONTRIBUTING.md §5.2` — добавить check в ревью миграций: «если добавляется таблица с `created_at` — рассмотреть партиционирование».
- Документация: `specs/01-data-model.md §8.10` — актуальная (уже написана под этот вариант).

## Применение к E1-E2

**E1 (инфраструктура):**
- E1.1 (docker-compose) — partitioning не затрагивается.
- E1.2 (первые модели Workspace, WorkspaceMember) — тоже не затрагивается (маленькие таблицы).

**E2 (core-модели):**
- Первая миграция с EstimateItem → создаём уже партиционированной.
- AuditLog → уже с RANGE partitioning по created_at.
- LLMUsage → уже с RANGE partitioning по created_at.
- Celery-beat task `maintain_partitions` — часть E2 или отдельный PR E2.1.

## Альтернативы, которые отвергли

1. **«Партиционировать только EstimateItem»** — compromise, но AuditLog/LLMUsage также страдают при 2М+ строк. Не экономит много работы.
2. **«Ждать и мониторить размер»** — риск пропустить границу, дорогая миграция потом.
3. **«Отдельная БД для логов»** — overengineering для MVP. Вернёмся к этому на этапе 3 (коробка/SaaS).

## Триггер пересмотра

Пересматриваем, если:
- Реальная нагрузка отличается от оценок в 10+ раз (например, клиент генерит 10М записей/год — нужен доп. уровень партиционирования).
- PostgreSQL выпустил фичу, упрощающую partitioning (unlikely в MVP window).
- Переходим на другую СУБД (unlikely).
