### #26 🟠 major — префикс порядкового номера в name

**Цитата:** «Если в первой Спецификации там были проектные обозначения систем типа "ВД1" и соединение их со столбцом Наименование через тире "-" — было полностью верно. Тут не так — соединение порядковых номеров с наименованием не нужно. Даже если это цифры вида "1.1" или "2.4" — соединять их с Наименованием не нужно.»

**Симптом в DB:** `name = "1.1 Комплект автоматизации…"`, `"1.2 Комплект…"`, …, `"2.1 Корпус…"`, `"3.1 Кабель…"`.

**Корневая причина:** LLM в промпте (правило 5 «Префикс-колонка») не различает алфавитно-цифровой код системы («ПВ-ИТП», «ВД1») от чистого порядкового номера («1.1», «2.4»). Склеивает всё подряд через `-` / пробел.

**Severity:** 🟠 major (косметически мешает, но всё редактируемо).

**Root cause:** **R21** — Numeric vs system code detection в промпте.

### #27 🔴 blocker — shift колонок для items 1-15 (model/brand/unit/qty)

**Цитата:** «Позиции с 1 по 10 — не определились единицы измерения. В Спецификации чётко прописано "шт.", в смету попало "1.0" на все десять позиций. Также на все 10 позиции неверно определена модель. Вместо модели взят столбец "Завод-изготовитель".»

**Симптом в DB (items 1-10):**
```
name = "1.1 Комплект автоматизации…"
unit = "1.0"              ← должно быть "шт."
quantity = 1.0000         ← случайно совпало
tech_specs.brand = "шт."  ← должно быть "ООО КОРФ"
tech_specs.model_name = "ООО КОРФ"  ← должно быть пусто
```

Для items 12-15 шифт повторяется (unit="255.0" вместо "м", brand="м" вместо "КАБЕЛЬ", model="КАБЕЛЬ\"" вместо пусто).

**Диагностика:** прогон `extract_structured_rows(page1)` для тех же row'ов **возвращает корректный маппинг**:
```json
{
  "cells": {
    "name": "1.1 Комплект автоматизации для приточной установки П1 в комплекте с П1",
    "brand": "ООО \"КОРФ\"",   // ← правильно
    "unit": "шт.",              // ← правильно
    "qty": "1,00",              // ← правильно
    "comments": "-учтено разделом ИОС4"
  }
}
```

→ **Bbox-парсер корректен, баг в LLM**. LLM «интеллектуально» переставляет поля когда видит отсутствующий `cells.model` — берёт `cells.brand` как `model_name`, `cells.unit` как `brand`, `cells.qty` как `unit`. Циркулярный shift.

Дополнительный фактор — LLM non-determinism: мой свежий curl-прогон вернул **корректный** JSON с `unit="шт."` и `model_name=""`, тогда как Андреев ранний прогон дал шифт. `temperature=0` не гарантирует детерминизм OpenAI.

**Severity:** 🔴 blocker (15 из 29 позиций с полностью неверными Ед.изм., Модель, Brand).

**Root cause:** **R19** — жёсткое правило в NORMALIZE_PROMPT: «не переставляй поля, копируй cells 1:1».

**Диагностика (`extract_structured_rows(page1).rows[7]`):**

{

### #38 🟠 major — лишние пробелы внутри чисел/слов: «Pc=3 0 0 Па», «Ø 2 25»

**Цитата:** «Вставляются лишние пробелы в наименование или модель. "Pc=3 0 0 Па" вместо "Pc=300 Па". Аналогично в строках 8-13 в модели.»

**Диагностика:** PDF использует отдельные text spans для цифр с мелким kerning. `extract_lines` в `pdf_text.py` склеивает все spans в одной y-row через `" ".join(...)`. Не проверяется x-gap.

**Severity:** 🟠 major (портит отображение, ломает fuzzy-matching моделей).

**Root cause:** **R24** — span-join с gap-threshold: если `span_i.x1 + font_size*0.3 >= span_{i+1}.x0` → склеивать **без** пробела.

### #39 🔴 blocker — пропуск 9 items между стр.3 и стр.4, плюс 4 items на стр.5

**Цитата:** «Между 3-й и 4-й страницей пропущено сразу 9 строк! На 4 странице первая строка сразу — "К1, К1.1 Внешний блок MDOAG-24HN8" — пропущено всё что выше на странице!»

### #40 🟡 minor — numeric prefix «1.1», «2.3» прилепляются к name в разделе БТП

**Цитата:** «Весь следующий раздел — номера типа "1.1", "2.3" прилеплены к наименованию, хотя мы кажется фиксили это?»

**Диагностика:** R21 в it1 закрывает **numeric prefix в `cells.name`** («1.1 Комплект автоматизации»). Но в ТАБС для раздела «Блочный тепловой пункт» префикс «1.1» попадает в `cells.pos` как системный код. Правило 5 промпта: «pos содержит буквы/не-цифры → alphanumeric → склеить через `-` к name». `1.1` — не чисто цифра (`.` допустим) — и LLM колеблется.

Дополнительно: если bbox column detection сломан (R23), «1.1» может попасть в `cells.name` → R21 не запускается для этой row.

**Severity:** 🟡 minor (косметически).

**Root cause:** **R21 уточнение** — явно формулировать что `\d+(?:\.\d+)*` это numeric **даже если** содержит точку.

### #44 🔴 стратегическое — мультимодальный LLM (JSON + image)

**Идея Андрея:** «Что если парсить без модели все данные с листа в JSON и отправлять потом модели эти данные вместе с картинкой листа?»

**Решение:** ✅ одобрено (гибрид). Реализуется как **R27** — conditional Vision fallback:

2. Confidence score страницы после phase 1.
3. Если confidence ниже порога → Phase 2: multimodal retry (JSON + rendered PNG) на той же странице.

**Принципы Андрея:** 
- Качество 100% для любых документов (включая Счета и КП).
- Скорость не критична — async workflow (колокольчик-notification).
- Цена не критична — дешевле работы сметчика.

→ **Смена модели:** extract переходит на **gpt-4o full** (точнее на сложных таблицах); classify остаётся gpt-4o-mini (дёшево).
